# Apex 1.18.0

The foundation for worker movement, built on the shared fleet.

## What changed for users

- **Pickups linked to housing.** A route stop can now point to a worker's **accommodation
  building**, so a pickup is tied to where workers actually live.
- **Trip boarding records.** A new **Trip Start Log** captures a trip's execution: who
  boarded, at which stop, by QR or manual check-in — counting expected vs boarded, and
  supporting unregistered workers (contractors and temporary hires) alongside registered
  ones.
- **One call for a driver's day.** A driver can fetch today's worker route — its ordered
  stops, each stop's housing pickup, and the worker manifest — from a single, identity-scoped
  endpoint. This is the backend that the worker-movement screens will build on next.

## What changed under the hood

- New submittable `Trip Start Log` + `Trip Boarding Event` child (derived boarded/expected
  counts, no GL), a nullable `accommodation_building` link on `Route Stop`, and a read-only
  `salis.api.masar.get_my_worker_route_today()` endpoint that resolves the driver server-side.
- Project row-scoping is wired through the trip's route plan, consistent with the other
  Movement documents.

## If you are updating an existing site

Pull and run the database migration; the new DocTypes install automatically and the new
route-stop field is optional, so existing data is unaffected.
