# Apex 1.11.0

Fuel requests now follow a proper approval workflow.

## What changed for users

- **One clear approval path for all fuel.** Every fuel request — a standard order, a quota
  top-up, or a fuel-chip action — now moves through **Pending → Approved → Done**, with a
  clear status at each step and a clean **Reject** path back. No more silent status edits.
- **Separation of duties.** The person who raises a fuel request can no longer approve it.
  Approval is reserved for fleet managers, so requests get a genuine second pair of eyes.
- **Quotas stay accurate.** Approving a standard request applies its fuel quota at the
  right step, and cancelling reverses it — so a vehicle's allowance always reflects what
  was actually approved.
- **The Fuel Approval Console keeps working.** The approve/reject console now drives the
  same workflow under the hood, so what you approve there and what the document shows are
  always in step.

## What changed under the hood

- `Fuel Request` adopts a native Frappe Workflow (the seventh document on the Salis
  approval spine), reusing its `status` field as the workflow state with self-approval
  disabled and request-type-aware transitions.
- The approval console was reconciled to act through the workflow rather than writing
  status directly, and the quota side-effect now applies on the post-submit transition to
  Done.

## If you are updating an existing site

Pull and run the database migration; the Fuel Request workflow is seeded automatically.
Existing requests keep their current status and continue from there.
