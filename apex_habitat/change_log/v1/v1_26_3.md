# Apex 1.26.3

A clearer split between the two kinds of transport, plus reliability fixes for how
records seed onto existing sites.

## What changed for users

- **Worker trips vs representative trips are now distinct.** A Transport Request
  on the Representatives line names the representative (an Employee) and can never
  be linked to labour accommodation or carry a worker manifest; a worker (bus /
  shuttle) trip cannot name a representative. Only worker trips appear in the
  worker (Masar) view.
- **A representative is not housed by the company.** The model no longer ties a
  representative to labour accommodation in any way.
- **Tidier Employee connections.** The fleet connections group on the Employee
  form now reads simply "Salis".

## What changed under the hood

- Added a `representative` (Employee) link to Transport Request with
  service-line-aware mandatory + separation guards in the controller; the Masar
  worker API now filters strictly to the Workers service line.
- Seed reliability: the Habitat kanban, assignment-rule, email-template and
  auto-email-report seeds now also run on `after_migrate` (they were install-only,
  so new records never reached existing sites) — matching the Salis seeds. (A
  follow-up will harden a couple of seed error-handling paths flagged by an audit.)

## If you are updating an existing site

- Run `bench migrate` (adds the `representative` column and reconciles the Habitat
  seeds), then `bench build` and `bench clear-cache`.
